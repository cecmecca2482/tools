Daily & Weekly Routine

A practical structure to keep learning from becoming abstract or overwhelming. Two to three focused hours a day compounds fast.

Daily Rhythm

  • Morning (60–90 min): Study one subtopic from a course or article — e.g., "RAG basics," "MLOps monitoring," "how vector DBs work."
  • Afternoon (60–90 min): Hands‑on play — build a tiny prototype, e.g., "AI‑powered PIM assistant in a no‑code tool" or "LLM + Airtable + webhooks."

Weekly Mini‑Project — Ship One Artifact Per Week

The "Map Back" Habit

As you learn each new concept, ask: "How does this map to my background?"

  • RAG → PIM as source of truth for LLM context.
  • Model drift monitoring → data‑quality SLAs you already know how to write.
  • Workflow orchestration → Boomi transformation maps, just AI‑native.
  • Model governance → data governance policies, applied to models instead of attributes.

Portfolio Project Ideas (Phase 4)

  • AI‑powered PIM assistant: A no‑code prototype that lets you ask questions about product data and get RAG‑backed answers.
  • Auto‑tagging workflow: A small pipeline that classifies product descriptions into taxonomy categories using an LLM.
  • AI governance spec: A written "AI‑by‑design" review document — showing you can define risk, lineage, fallbacks, and audit.
Visibility Tip

Post one thing per week on LinkedIn about what you're learning. Frame it as "I'm building an AI learning map tailored to product data and PIM — here's what I learned about RAG this week." It signals the exact combination of skills you're building.

Bridge Curriculum

This is not about becoming an engineer. It's about accumulating enough building scar tissue that you can work confidently inside the same tools engineers use. Think of it as learning to drive, not learning to build a car.

Core Principle

Engineers have an advantage in how to build. You have an advantage in what to build and why. This curriculum closes the "how" gap just enough that you become dangerous — and that combination is rarer and more valuable than either skill alone.

Stage 1 — Learn to Read the Engineer's World (Weeks 1–2)

GitHub — Demystified

GitHub is just a version-controlled cloud folder where code lives, with a social layer on top. That's it.

Week 1 action: Create a free GitHub account. Find 3 repos related to AI tools you've read about (search "LangChain", "n8n", or "open-source PIM"). Read their READMEs. You're not touching code — just navigating the environment.

The Terminal — One Small Crack in the Door

The terminal is just a text-based way to talk to your computer. You need to be able to open it, run a command someone gives you, and read the output without panicking.

Your First Terminal Command
pwd ← Mac: shows where you are on your computer cd ← Windows: same thing ls ← Mac: lists files in current folder dir ← Windows: same thing

VS Code — The Engineer's Notebook

VS Code is a free text editor where engineers write and run code. It's also where AI‑assisted building happens (GitHub Copilot, Claude Code). Action: Download VS Code, open a folder, create a file called notes.md, type some text. You've just used the same tool most engineers work in every day.

Stage 2 — Build Your First Real Thing (Weeks 3–4)

Project: AI‑Powered Product Description Generator

Build a workflow that takes a product name + a few attributes, sends them to an LLM, and returns a polished product description. This maps directly to your PIM background.

  • Tool to use: n8n or Zapier — no code required, but they expose you to the same logic engineers use: triggers, actions, inputs, outputs, conditionals.
  • When it breaks (it will): Read the error message slowly before Googling. This is the most important engineering habit you can build.

Hands‑On with APIs via Postman

Download Postman (free). Make a real API call to the OpenAI API. Send a prompt, get a response. You've just done exactly what every AI application does under the hood — you're seeing the raw version of it. This demystifies "calling an API" permanently.

Stage 3 — Learn to Read Code, Not Write It (Weeks 5–6)

Python Basics — Just Enough

Python is the dominant language in AI. You need to read a Python script and understand what it's doing at a high level — not write it yourself.

  • What to study: learnpython.org — do the first 5 lessons only: Hello World, Variables, Lists, Basic Operators, String Formatting. Stop there.
  • The practice habit: Once a week, paste a short Python script into Claude and ask "explain what each section of this code does in plain English." Do this until code stops feeling like a foreign language.

JSON — The Language AI Systems Use to Talk to Each Other

Most AI APIs send and receive data in JSON. If you've worked with PIM data exports, you've seen it.

JSON is just labeled data — that's it
{ "product_name": "Ergonomic Chair", "color": "Black", "in_stock": true, "attributes": { "weight_capacity_lbs": 250, "material": "Mesh + Nylon" } }

Action: Go to jsonlint.com. Paste some JSON. Edit it. Break it and fix it. JSON fluency is a genuine superpower when working with AI systems.

Stage 4 — Build with AI as Your Coding Partner (Weeks 7–8)

Install Claude Code or GitHub Copilot

Claude Code and GitHub Copilot are AI coding assistants inside VS Code. They write code based on your plain‑English instructions.

Key Mindset Shift

You are the product manager of the AI that is writing your code. You write the spec in plain English, the AI writes the implementation, you QA it. This is a skill you already have — you're just applying it to a new domain.

Capstone Project: RAG‑Powered PIM Q&A Tool

  • Create a CSV with 20–30 fake product records (use your PIM knowledge — SKUs, titles, categories, attributes, compliance flags).
  • Use Claude Code or Copilot to write a Python script that loads that data, creates embeddings, stores them in a simple vector store, and lets you query it with natural language.
  • You will not understand every line of code. That's fine. Focus on: does it work? Does it return accurate answers? What breaks it?
  • Document what you built in a README on GitHub. This is now a portfolio artifact.

Bridge & Tools

AI‑Engineering Thinking

How to think like an engineer who uses AI — without writing code. These mental models are more valuable than syntax.

1 — Think in Systems, Not Tools

  • AI is a service/module in a larger system: it has inputs, outputs, latency, cost, and failure modes.
  • Always ask: Where does AI sit in the data‑flow? (PIM → AI → UI / ERP / Workflow Engine).

2 — Decompose Workflows into AI‑Friendly Steps

  • Data inputs (PIM, ERP, tickets, dashboards).
  • Decision points (classify, rank, summarize, QA).
  • Action points (update PIM, send email, route to human).
  • Map AI only to routine, data‑heavy steps; keep high‑risk decisions human‑driven.

3 — Treat AI Like an "API" with Contracts

For every AI module, define:

  • Inputs — what data, where it lives.
  • Outputs — format, structure, confidence score.
  • Constraints — latency, PII rules, accuracy thresholds, fallbacks.

4 — Design for Failure

  • AI will be wrong sometimes. Define: what happens when confidence is low? Route to human? Use cached result? Return default?
  • Every AI step needs a fallback path — no exceptions.

5 — Monitor Like an Engineer

  • Track: accuracy over time, latency, hallucination rate, user correction rate, data drift.
  • Set SLAs for each AI module just like you'd set SLAs for an API.

6 — Governance Is a Feature, Not a Tax

  • Who owns the data quality feeding this model?
  • Who approves model updates?
  • Who is notified when accuracy drops?
  • Is there an audit log?
Summary Mental Model

Input contract → AI module → Output contract → Fallback → Monitor → Govern.

AI‑Vibe‑Coding PIM Examples

What "AI‑vibe‑coding" looks like when applied to PIM workflows — plain‑language prompts that produce real, structured outputs.

Example 1 — Auto‑Classify a Product into a Taxonomy

PROMPT: "Given this product title and description, classify it into the correct category from our taxonomy. Return category path, confidence score, and reason." INPUT: { title: "Wireless Noise‑Cancelling Headphones", description: "Over‑ear, Bluetooth 5.2, 30hr battery..." } Taxonomy: Electronics > Audio > Headphones > Over‑Ear OUTPUT: { category: "Electronics > Audio > Headphones > Over‑Ear", confidence: 0.97, reason: "Over‑ear, wireless, audio device matches taxonomy node." }

Example 2 — Generate Missing Product Attributes

PROMPT: "Given this product data, identify missing required attributes and suggest values based on the product type and description." INPUT: { title: "Ergonomic Office Chair", description: "Adjustable lumbar support, mesh back, 5‑star base..." } Required attributes: color, weight_capacity_lbs, material, adjustable_height OUTPUT: { color: "Black (inferred from image alt‑text)", weight_capacity_lbs: "250 (industry standard for this category)", material: "Mesh + Nylon", adjustable_height: "Yes (mentioned in description)" }

Example 3 — QA Flag for Compliance‑Critical Attributes

PROMPT: "Review this product record for compliance gaps. Flag any missing or suspicious values for regulated attributes." INPUT: { product_id: "SKU‑4821", category: "Children's Toys", age_rating: null, material: "Unknown", safety_cert: null } OUTPUT: { flags: [ { attribute: "age_rating", issue: "Missing – required for children's toys" }, { attribute: "material", issue: "Unknown – required for CPSC compliance" }, { attribute: "safety_cert", issue: "Missing – ASTM F963 cert required" } ], risk_level: "HIGH" }

Example 4 — Summarize a Product for a Channel

PROMPT: "Rewrite this product description for a mobile‑first e‑commerce channel. Max 80 words, lead with the top benefit, end with a CTA." INPUT: Long technical spec description (300 words)... OUTPUT: "Crystal‑clear sound meets all‑day comfort. These over‑ear headphones deliver 30 hours of noise‑cancelling audio on one charge — perfect for work, travel, or anything in between. Bluetooth 5.2 keeps you connected instantly. Shop now and hear the difference."

Example 5 — RAG‑Grounded Product Q&A

SETUP: PIM data for 500 SKUs loaded into a vector DB (e.g., Pinecone). PROMPT: "Is the Model X standing desk compatible with dual‑monitor arms from third‑party vendors?" WITHOUT RAG: "Most standing desks support dual‑monitor arms..." [generic, potentially wrong] WITH RAG: "According to the Model X spec sheet (SKU‑1042), the desk has a grommet hole diameter of 50mm and a max monitor arm load of 20kg per side. It is compatible with any VESA‑standard arm meeting those specs." [grounded in your actual PIM data]

MCP & AI Integrations

Model Context Protocol is the most important new infrastructure topic in AI right now — and your Boomi/integration background makes you uniquely positioned to understand and talk about it. This is not optional knowledge in 2026.

Why This Matters Right Now

MCP launched in November 2024 and has already become the universal standard for connecting AI agents to enterprise tools — with 97M+ monthly SDK downloads and adoption by OpenAI, Google, Microsoft, and AWS. Forrester predicts 30% of enterprise SaaS vendors will launch their own MCP servers in 2026. If your product doesn't have one, it's invisible to AI agents.

What MCP Actually Is

Before MCP, every AI agent that needed to access external data — a PIM system, a CRM, a database, a ticketing tool — required a completely custom integration. Each vendor built their own approach. The result was the same fragmentation problem that REST APIs solved a generation earlier: a many-to-many integration mess with no shared standard.

MCP fixes this. It's an open standard — often described as "USB-C for AI" — that defines how any AI model connects to any external tool or data source using a single, standardized interface. An AI can connect to your file system, your PIM, your ERP, GitHub, Slack — and actually act on them, not just talk about them.

  • MCP Host: The AI application (e.g., Claude, ChatGPT, your product) that wants to use tools.
  • MCP Client: The component inside the host that speaks the MCP protocol.
  • MCP Server: A lightweight service that exposes a tool, data source, or system (e.g., "Salsify PIM server," "Jira server," "your database server") in a standardized way the AI can discover and use.

The Boomi Connection — Why This Is Your Mental Model

You've worked with Boomi to transform and map data from point A to point B using standardized connectors. MCP is that same idea, except the AI model itself becomes an actor in the integration — it doesn't just pass data through, it can read, write, query, and trigger actions across systems based on a goal you've given it. Think of MCP servers as Boomi connectors, but for AI agents.

Subtopics to Learn

Key Concepts You Must Own

  • MCP is now governance-neutral infrastructure — donated to the Linux Foundation's Agentic AI Foundation (AAIF) in Dec 2025, co-governed by Anthropic, OpenAI, Google, Microsoft, and AWS. It's not going away.
  • For a PM, MCP changes your product's surface area — your product now needs an MCP strategy just like it needs an API strategy. "Is our product MCP-ready?" is a real product roadmap question.
  • MCP servers expose three things to AI agents: Tools (things the AI can do), Resources (data the AI can read), and Prompts (templates for common tasks). As a PM you define all three.
  • Security and permissions are a product design problem, not just an engineering one — you define what an AI agent is and isn't allowed to do in your system.

Hands-On Actions

  • Browse the official MCP servers repo on GitHub — look at what's already been built (Slack, GitHub, databases, file systems). Read the READMEs.
  • Try Claude Desktop with an MCP server connected (the filesystem server is the easiest starting point) — this lets you see an AI actually reading and acting on your local files via MCP.
  • Write a 1-page "MCP strategy memo" for a hypothetical PIM product: what tools, resources, and prompts would you expose? What permissions would you lock down? This is a portfolio artifact.

Resources

  • modelcontextprotocol.io — the official spec and documentation. Skim the overview; you don't need to read the full spec.
  • Truto — "What is MCP: The 2026 Guide for SaaS PMs" — written specifically for product managers.
  • Imagine Works — "MCP: A Guide for Enterprise Leaders" — covers the procurement and governance angle.

Emerging · 2026

AI Agents & Multi‑Agent Systems

Agents stopped being toys in late 2025. By early 2026 they're reliable in scoped enterprise workflows. Understanding how to design for, govern, and evaluate agentic systems is rapidly becoming a core PM skill.

The Shift That Happened

For most of 2024, AI agents were impressive demos that broke in production. By late 2025, hallucinations dropped, planning improved, tool use matured, and agents became genuinely reliable in scoped workflows. The question has shifted from "do agents work?" to "how do we design workflows for them responsibly?"

What Is an AI Agent, Really?

An agent is an AI system that can take a goal, break it into steps, use tools to accomplish those steps, observe the results, and iterate — without a human approving each action. The key properties that make something truly agentic:

  • Autonomy: Acts independently without continuous human oversight.
  • Tool use: Can call APIs, read files, write to databases, send messages.
  • Memory: Can remember context across steps, sessions, or time.
  • Reactivity: Adapts its plan when it gets unexpected results.
  • Proactivity: Initiates actions to achieve a goal, not just responds.

Multi-Agent Systems

A multi-agent system is when multiple AI agents collaborate — each specializing in a domain, handing off work, and checking each other's output. Think: an "orchestrator" agent that plans and delegates, a "researcher" agent that gathers information, a "writer" agent that drafts content, and a "reviewer" agent that checks quality. MCP is the plumbing that lets these agents talk to each other and to external tools.

Subtopics to Learn

Key Concepts You Must Own

  • Most agent failures are intent failures, not model failures — the agent didn't understand the goal well enough. Your job as PM is to define intent precisely.
  • Design human-in-the-loop checkpoints deliberately — not as a fallback, but as a first-class product feature. Define which decisions an agent can make autonomously and which require human approval.
  • In enterprise settings, agents need the same governance treatment as any other system: audit logs, permission boundaries, rollback plans, and escalation paths.
  • The value of multi-agent systems is parallel specialization — different agents handle different expertise domains simultaneously, then synthesize. This is where PIM + AI gets very interesting (enrichment agent, QA agent, compliance agent all working on the same product record).

PIM-Specific Agent Design Idea

Example: Multi-Agent PIM Enrichment Pipeline
Orchestrator Agent: "Enrich this batch of 500 new SKUs" → Research Agent: scrapes manufacturer specs, pulls category standards → Enrichment Agent: fills missing attributes, suggests values with confidence scores → Compliance Agent: checks regulated attributes (age ratings, safety certs, materials) → QA Agent: flags low-confidence fills and compliance gaps for human review → Human checkpoint: reviewer approves or corrects flagged items → Write Agent: commits approved enrichments back to PIM

Resources

  • LangGraph documentation — visual multi-agent workflow design (read for the concepts, not the code).
  • Anthropic's "Building effective agents" guide — the clearest conceptual overview of agentic patterns.
  • Search "agentic AI product management 2026" for current practitioner writing.
Emerging · 2026

AI Evals (Evaluations)

Called the single most important new skill for AI product managers in 2026. AI features don't fail because of the model — they fail because nobody defined what "good" meant and tested for it systematically.

The Core Problem Evals Solve

When you ship a traditional feature, you can write deterministic tests: "if input X, expect output Y." AI features don't work that way — outputs are probabilistic. Evals are the framework for systematically measuring whether your AI feature is working well enough, consistently enough, across enough cases, to be in production.

What Are Evals?

An "eval" is an evaluation — a test suite designed specifically for AI outputs. Instead of checking if code returns the right value, you're checking if an LLM returns a good enough answer across a representative set of real-world inputs. Evals answer questions like:

  • Does our product description generator produce accurate, on-brand output for 95%+ of inputs?
  • Does our PIM classification agent get the taxonomy right for the edge cases, not just the easy ones?
  • Did our new prompt version perform better or worse than the old one — and can we prove it?
  • Are there categories of input where the model consistently fails?

Subtopics to Learn

Key Concepts You Must Own

  • Start with evals, not instructions. Before you write a prompt, define what "good output" looks like and how you'll measure it. This reverses how most teams work — and it's the right order.
  • LLM-as-judge is a practical pattern: use a second, more capable LLM to evaluate the outputs of your production LLM at scale. You define the rubric; the judge applies it to thousands of outputs automatically.
  • An eval dataset is a living product artifact — it should grow every time a new failure mode is discovered in production. You own this, not engineering.
  • Evals plug directly into your data governance instincts: who defines the rubric? Who maintains the dataset? Who reviews when scores drop? These are governance questions you already know how to answer.

A PIM Eval Example

Simple eval rubric for a product description generator
For each generated description, score 1–5 on: · Accuracy: Does it match the source spec data? No hallucinated attributes? · Completeness: Does it include all required attributes for this category? · Tone: Does it match brand voice guidelines? · Length: Is it within the target word count for this channel? · CTA: Does it end with an appropriate call to action? Passing threshold: average score ≥ 4.0, no Accuracy score below 3. Run automatically on a 50-sample dataset every time the prompt changes.

Emerging · 2026

Context Engineering

As models get better and cheaper, the competitive edge shifts away from which model you use — and toward how well you architect the context around it. Context engineering is the new prompt engineering, and it's a PM-level skill.

The Key Insight from AI Engineer Europe 2026

"Models are good enough. What is lagging is everything around them: evaluation frameworks, context engineering, governance, domain expertise, and organisational design. The teams who build that infrastructure now will compound the advantage over the next 2–3 years."

What Is Context Engineering?

Prompt engineering was about crafting the right words in a single prompt. Context engineering is the broader discipline of designing everything that goes into an AI model's context window — not just the prompt, but the system instructions, retrieved documents (RAG), conversation history, tool outputs, user state, and memory — to reliably produce the behavior you want across the full range of real-world inputs.

It's the difference between "write a good prompt" and "design an AI system that consistently behaves well."

The Components of Context

  • System prompt / instructions: The persistent instructions that shape the model's behavior, persona, and constraints across all interactions.
  • Retrieved context (RAG): The documents, records, or data pulled from your knowledge base and injected into the context based on the user's query.
  • Conversation history: Prior turns in the conversation — how much to include, how to summarize long histories.
  • Tool outputs: Results from tools the model has already called (e.g., a database query result, an API response).
  • User state & metadata: Who is the user? What permissions do they have? What product are they viewing?
  • Examples (few-shot): Carefully chosen input/output examples that demonstrate the behavior you want.

Subtopics to Learn

Key Concepts You Must Own

  • Context quality is a data quality problem — and you already know how to think about data quality. Bad context in → bad model behavior out, regardless of how good the model is.
  • The most common agent failures are intent failures: the agent had the right tools but the wrong understanding of the goal. Fix this with better strategic context, not more instructions.
  • Context is a product design decision, not just an engineering one. What information does this AI need at this moment to behave the way our users need it to? That's a PM question.
  • For PIM specifically: the context you inject (product category standards, brand voice guidelines, compliance rules, channel requirements) is what transforms a generic LLM into a domain-expert PIM assistant.

PIM Context Engineering Example

What goes into context for a PIM description generator
SYSTEM PROMPT: You are a product content specialist for [Brand]. You write accurate, compelling product descriptions that follow brand voice guidelines. Never invent specifications not present in the source data. Always flag low-confidence fills with [REVIEW NEEDED]. RETRIEVED CONTEXT (RAG): · Category standards for "Over-Ear Headphones" (pulled from PIM taxonomy) · Brand voice guide excerpt relevant to audio products · 3 example descriptions for similar approved products (few-shot) USER INPUT: · Raw product spec data for SKU-1042 · Target channel: mobile e-commerce · Required attributes for this category RESULT: A grounded, on-brand description — not a hallucinated one.
